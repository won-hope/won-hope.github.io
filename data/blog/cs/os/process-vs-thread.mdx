## 💡 Introduction: 기초 CS가 아키텍처의 기준이 되는 순간

**"동시 접속자가 10배 늘어나면, 서버의 `max-threads`도 10배로 늘리면 해결될까?"**

백엔드 개발을 학습하며 가장 많이 듣는 단어 중 하나가 '동시성(Concurrency)'이다. 트래픽이 몰릴 때 서버가 버티지 못하고 다운되거나 메모리 부족(OOM)을 겪는 이유는 무엇일까? 

이를 이해하려면 대학 전공 서적 첫 장에 나오는 프로세스(Process)와 스레드(Thread)의 개념을 'OS 관점'이 아닌 '서버 애플리케이션 관점'에서 다시 바라봐야 한다.

면접에서 흔히 묻는:
> "프로세스와 스레드의 차이가 뭔가요?"

라는 질문은 단순히 자원 공유 여부를 외우고 있는지 확인하기 위함이 아니다. 
"이 지원자가 서버 자원의 한계를 이해하고, 병목 현상의 원인을 아키텍처 관점에서 추론할 수 있는가?"를 확인하는 핵심 질문이다.

단순한 정의를 넘어, 이 개념이 실제 Spring Boot 환경에서 어떻게 동작하고 어떤 방향성으로 시스템을 설계해야 하는지 정리해 보았다.

## 🏗️ Body: CS 개념을 Spring Boot 환경으로 번역하기

### 1. Process vs Thread: 공장과 작업자

'자원 공유 여부'라는 딱딱한 텍스트 대신, 직관적인 비유로 접근해 보자.

- **🏭 Process (프로세스): 하나의 독립된 공장(Factory)**
  우리가 `java -jar app.jar`를 실행해 띄운 Spring Boot 애플리케이션 자체가 하나의 프로세스다. OS로부터 땅(메모리)을 할당받아 지어진 공장이며, 공장끼리는 두꺼운 벽이 쳐져 있어 서로의 자재(메모리)를 함부로 건드릴 수 없다.

- **👷 Thread (스레드): 공장 안에서 일하는 작업자(Worker)**
  작업자들은 공장 안의 자재와 기계(Heap 영역)를 자유롭게 공유하며 일한다. 하지만 각자 자신이 지금 무슨 작업을 어디까지 했는지 적어두는 개인 수첩(Stack 영역)은 따로 가지고 있다.

---

### 2. Spring Boot와 Tomcat: 작업자 대기실 (Thread Pool)

Spring Boot(Spring MVC)는 기본적으로 내장 Tomcat을 사용하며, **Thread per Request** 모델로 동작한다. 즉, HTTP 요청이 하나 들어올 때마다 전담 작업자(스레드) 한 명이 배정되어 처음부터 끝까지 책임을 진다.

하지만 요청이 올 때마다 작업자를 새로 채용(생성)하고 해고(소멸)하는 비용은 OS 입장에서 매우 크다.

> **💡 Thread Pool의 존재 이유**
> 이를 해결하기 위해 Tomcat은 미리 200명(기본값)의 작업자를 대기실(Pool)에 모아둔다. 요청이 오면 대기실에서 한 명씩 내보내서 일하게 한 뒤, 일이 끝나면 다시 돌려보낸다. 이는 스레드 생성 비용을 아끼고, 한 번에 처리할 수 있는 요청 수를 제어하는 훌륭한 방파제 역할을 한다.

---

### 3. 🤔 가설 검증: 스레드를 무작정 늘리면 어떻게 될까?

트래픽이 평소 대비 10배 몰린다고 가정해 보자. 다음과 같이 대기실의 크기(`max-threads`)를 10,000개로 넉넉하게 잡으면 서버가 10배의 일을 해낼 수 있을까?

```yaml
server:
  tomcat:
    threads:
      max: 10000
```

CS 지식을 기반으로 분석해보면, 겉보기엔 그럴싸하지만 실제로는 세 가지 치명적인 문제가 발생한다.

#### 🚨 문제 1: 메모리 폭발 (OOM, Out Of Memory)
스레드는 각자의 '개인 수첩(Stack)'을 가진다고 했다. Java에서 스레드 하나를 생성하면 기본적으로 약 **1MB의 Stack 메모리**를 할당받는다. 
즉, 스레드가 10,000개라면 순수하게 스레드 유지에만 **10GB의 물리적 메모리**가 필요하다. 일반적인 클라우드 인스턴스 환경이라면 즉시 OOM 에러를 발생시키며 프로세스 자체가 뻗어버릴 것이다.

#### 🚨 문제 2: Context Switching의 저주 (Thrashing)
사장님(CPU 코어)은 소수인데, 관리해야 할 작업자(스레드)가 1만 명이라면?
사장님은 A 작업자의 수첩을 보다가, B 작업자로, 다시 C 작업자로 쉴 새 없이 시선을 돌려야 한다. 이를 Context Switching(문맥 교환)이라고 한다.

이전 작업 상태를 저장하고, CPU 캐시를 비우고, 새로운 상태를 로드하는 것은 매우 무거운 작업이다. 결국 1만 개의 스레드를 번갈아 살피느라 자원을 다 쓰고, 정작 실제 로직은 하나도 처리하지 못하는 Thrashing(스레싱)에 빠진다. 서버는 켜져 있지만 응답을 하지 못하는 Hang 상태가 된다.

#### 🚨 문제 3: 외부 자원 병목 (DB Connection Pool 고갈)
가장 흔하게 겪는 실무 장애다. Tomcat 스레드를 1만 개로 늘렸다고 해서 DB도 1만 개의 요청을 동시에 받아주는 것은 아니다. 

Spring Boot는 주로 HikariCP 같은 DB Connection Pool을 사용하는데, 기본 커넥션 수는 보통 10개 내외다. 
결국 1만 개의 스레드가 생성되어도, 나머지 9,990개의 스레드는 DB 커넥션을 얻기 위해 무한정 대기(Blocking)하게 되고 결국 Timeout 에러를 뿜어내며 시스템 전체가 마비된다.

## 🛠️ Direction: 올바른 아키텍처 설계 방향성

위의 분석을 통해, "단일 애플리케이션 내에서 스레드를 늘리는 것만으로는 트래픽 문제를 해결할 수 없다"는 결론에 도달한다. 그렇다면 어떤 방향성으로 아키텍처를 설계해야 할까?

1. **Process의 확장 (Scale-out)**
   단일 공장(Process) 안에서 작업자를 한계치 이상으로 쥐어짜는 대신, **공장 자체를 여러 개 복제하고 Load Balancer로 트래픽을 분산**시킨다. 가장 표준적이고 안전한 구조다.

2. **작업 방식의 패러다임 전환 (Non-blocking / Asynchronous)**
   Thread per Request 방식의 한계를 극복하기 위해, 소수의 스레드가 DB 조회 등의 I/O 작업을 기다리지 않고(Non-blocking) 계속 다른 요청을 처리하게 만드는 **Spring WebFlux**나 Event Loop 기반의 아키텍처를 도입할 수 있다.

3. **트래픽 완충 지대 마련 (Message Queue)**
   모든 요청을 실시간으로 동시에 처리할 필요가 없다면, 앞단에 **Kafka**나 **RabbitMQ** 같은 Message Queue를 두어 트래픽을 안전하게 담아둔다. 서버는 자신이 감당 가능한 속도로만 이벤트를 소비(Consume)하게 만들어 시스템 전체를 보호한다.

## 🚀 Conclusion: CS는 'How'를 결정하는 'Why'다

Process와 Thread를 학습하며 깨달은 것은, 이 기초 지식들이 단순히 면접 통과를 위한 텍스트가 아니라는 점이다. 

"왜 이 프레임워크를 쓰는가?", "왜 이런 아키텍처를 선택했는가?"라는 질문에 대한 근본적인 해답은 결국 "메모리를 어떻게 관리하고 CPU 자원을 어떻게 분배할 것인가"라는 OS 레벨의 이해에서 출발한다.

이번 학습을 바탕으로, 다음 단계에서는 단순한 이론 학습을 넘어 **JMeter나 nGrinder 같은 성능 테스트 도구를 활용해 토이 프로젝트에 인위적인 부하를 주고, Thread Pool 사이즈 변화에 따른 성능 지표(Throughput, Latency)를 직접 눈으로 확인하고 튜닝해 보는 경험**을 쌓아볼 계획이다.
